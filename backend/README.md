## 1. Этапы бота

Сверху вниз: **бот только для админа** → **приём алертов** (HTTP от монитора) → **кнопки и 3x-ui**. Ниже — подъэтапы; по ним идём как по урокам: сначала кратко что и зачем, потом код для ручного переноса.

## 2. От создания до подключения

1. **Telegram** — `@BotFather` → `/newbot` → имя → username → **токен**.
2. **Проект бота** — отдельный сервис (папка / `cmd`), зависимости, `README` только с этим файлом как оглавлением.
3. **Секреты** — токен и `ADMIN_TELEGRAM_USER_ID` только из env / секретов, не в git.
4. **Запуск** — локально или контейнер: процесс слушает апдейты Telegram.
5. **Подключение** — админ пишет боту `/start` (или любую тестовую команду); бот отвечает только ему — связь считается рабочей.

## 3. Этапы и подъэтапы

### Этап A — Каркас и конфиг

- **A1** — структура репозитория бота (точка входа `main`, пакет `bot` или аналог).
- **A2** — загрузка конфига из env: `TELEGRAM_BOT_TOKEN`, `ADMIN_TELEGRAM_USER_ID` (при необходимости список через запятую).

### Этап B — Long polling и фильтр админа

- **B1** — клиент к Telegram Bot API, цикл long polling (webhook — позже, когда будет HTTPS).
- **B2** — middleware / проверка: обрабатываем только апдейты от `ADMIN_*`, остальное игнор.
- **B3** — команда `/start` (и опционально `/help`) — ответ только админу, проверка что бот «жив».

### Этап C — Внутренний HTTP для алертов

- **C1** — маленький HTTP-сервер на том же процессе (или второй порт): `POST /internal/alert` с секретом в заголовке.
- **C2** — тело: JSON (например `client_uuid`, `ip_count`, `max_ips`, текст); валидация → отправка сообщения админу в Telegram.
- **C3** — простая защита: `X-Internal-Token` или общий секрет с монитором.

### Этап D — Связка с VpnAPI (позже по урокам)

- **D1** — HTTP-клиент к `VpnAPI` с `INTERNAL_TOKEN` (получить клиента по UUID и т.д.).
- **D2** — форматирование алерта с данными из API.

### Этап E — Кнопки и 3x-ui (после готовности панели)

- **E1** — inline-кнопки, `callback_query`.
- **E2** — клиент 3x-ui, выдача ключа/ссылки по согласованному контракту.

## 4. Несколько VPN-серверов (control-plane)

- Панели 3x-ui хранятся в таблице `vpn_servers` (миграция `000003`).
- При старте VpnAPI строка `default` синхронизируется из `XUI_*` в `.env` (первый VPS).
- Второй и следующие узлы — `INSERT` в `vpn_servers` (см. `deploy/servers.example.sql`).
- API: `GET /api/v1/servers`, при создании клиента — `serverId` в `POST /api/v1/clients`.
- Provision/Revoke идут в 3x-ui того сервера, который записан у клиента.
- VpnAPI на control-plane VPS должен достучаться до `xui_base_url` каждого узла по сети.
- Вторая панель из env: блок `VPN_SERVER_VPS1_*` в `backend/.env` (см. `.env.example`) — при старте строка создаётся **один раз**, если `vps_1` ещё нет в БД.
